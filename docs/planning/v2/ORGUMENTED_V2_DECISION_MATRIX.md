# ORGUMENTED v2 DECISION MATRIX
Stage-Aligned Focus Matrix (March 2026+)

Purpose:
Channel focus toward Stage 1: Trusted Change Decision Engine
Prevent premature governance expansion.
Protect deterministic moat.

Scoring:
1 = weak / harmful
2 = risky
3 = neutral
4 = strong
5 = directly strengthens Stage 1 moat

Weighted Score = Score × Weight

---

## SECTION 1 — Deterministic Integrity (Weight: 30)

1.1 Replay parity protection  
Does this change preserve identical output identity under identical snapshot + query + policy?

1.2 Proof artifact integrity  
Does it strengthen, preserve, or weaken proof traceability?

1.3 Fail-closed clarity  
Does it preserve constraint refusal behavior with explicit reasoning?

1.4 Dev vs packaged parity  
Does it reduce or increase desktop/runtime divergence?

If Section 1 average < 4.0 → STOP.

---

## SECTION 2 — Planner / Compiler Depth (Weight: 20)

2.1 Moves beyond regex-heavy routing  
2.2 Increases ontology-aware reasoning depth  
2.3 Improves composability of decision packets  
2.4 Enables measurable improvement in blast-radius or risk reasoning

If planner depth stagnates → strategic claims degrade.

---

## SECTION 3 — Decision-Packet Workflow Fit (Weight: 20)

3.1 Packets usable without raw JSON inspection  
3.2 Reduces review-time evidence gathering  
3.3 Increases operator trust  
3.4 Fits real review or approval workflows

If workflow fit < 3.5 → do not expand into policy gating.

---

## SECTION 4 — Runtime Convergence (Weight: 15)

4.1 Reduces browser-era seams  
4.2 Strengthens single UI→engine boundary  
4.3 Improves desktop startup/smoke reliability  
4.4 Reduces standalone Next-server assumptions

---

## SECTION 5 — Measurable Lift (Weight: 10)

5.1 Can we measure improvement in:
- time-to-trusted-answer?
- blast-radius clarity?
- evidence-assembly effort?
- escalation reduction?

If measurable lift undefined → deprioritize.

---

## SECTION 6 — Strategic Stage Alignment (Weight: 5)

6.1 Strengthens Stage 1 directly  
6.2 Premature Stage 2/3/4 expansion? (penalize if yes)

---

# Hard Gating Rules

1. If Deterministic Integrity avg < 4.0 → reject.
2. If Planner Depth does not improve over current baseline → deprioritize.
3. If Workflow Fit is weak → do not expand policy support.
4. If change increases runtime divergence → reject.
5. No Stage 2/3 claims without Stage 1 proof.

---

# Evaluation Outputs Required

For any major initiative:

1. Completed matrix with numeric justification
2. Explicit note on which Stage it strengthens
3. Evidence plan for measurable lift
4. Risk of strategic overreach
5. Kill-switch trigger defined

---

# What This Matrix Is Not For

Not for:
- minor refactors
- small UI polish
- bug fixes

This matrix governs:
- planner/compiler changes
- policy engine changes
- workflow redesign
- governance expansion
- major runtime shifts
